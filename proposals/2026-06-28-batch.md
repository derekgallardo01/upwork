### Cover letter text — answer the 5 screening questions inline (don't make them search)

```
Hi <first name>,

The part of this that usually sinks "AI over a newsletter archive"
projects is link retention — the LLM paraphrases URLs into oblivion
and the whole point of the archive (the actual tools and articles you
were saving) is lost. My open RAG kit is built citation-first, and the
same enforcement pattern (system-prompt + post-process validation)
applies one-to-one to your newsletter pipeline:

  github.com/derekgallardo01/rag-over-docs-kit (live demo, eval harness)

For the multi-cluster routing and the human-in-the-loop "Review
Dashboard," the closest pattern in my work is the lead-triage kit —
deterministic classifier in front of the LLM, with a separate queue
for low-confidence cases that need a human eye before they're
auto-actioned:

  github.com/derekgallardo01/nocode-ai-lead-workflow

Direct answers to your 5 screening questions:

1. iCloud integration — IMAP via an Apple app-specific password
   (icloud.com → Sign-In and Security → App-Specific Passwords). A
   Python script using `imaplib` walks the folder structure and writes
   each email's raw text + headers + the original HTML into a SQLite
   intermediate. Zapier never sees the archive, which is what keeps
   the historical sweep affordable.

2. Multi-cluster, future-extensible — topic rules live in a single
   YAML/JSON registry: one entry per cluster with `label`,
   `sender_match` / `gmail_label_match` rules, `system_prompt`, and
   `output_schema`. Adding a new topic (say, Real Estate) is a YAML
   entry + a prompt — no automation rebuild, no breaking the existing
   ones.

3. Link retention — strict system-prompt rule ("Preserve every
   <a href> URL verbatim. Never paraphrase a URL. Wrap with markdown
   link syntax.") + a post-process regex validation step that confirms
   every URL in the source either appears in the output OR is
   explicitly noted as filtered (e.g. unsubscribe links). Failures
   route to the human-review queue, not silently dropped.

4. Backlog cost control — Anthropic batch API (50% off async)
   processing in batches of 500–1000 emails. Rough estimate: 2 years
   × ~30 newsletters/week × ~5 KB tokens each ≈ ~5M input tokens,
   ~$50–100 of API cost at Claude Haiku. Custom Python script for the
   sweep, not Zapier — that's the cost decision.

5. Airtable vs Notion — Airtable for this exact use case. Better
   relational primitives (linked tables across clusters, podcast
   episode planning, newsletter draft tracking), views/filters per
   cluster come native, and the human-in-the-loop status field +
   automation triggers are turnkey. Notion's relational model is too
   loose for the multi-cluster routing you've described.

How I'd run yours:

  Week 1 — paid 1-hr scoping call ($175, credited against the
  fixed price) to confirm: the 3 (or 4–5) initial clusters, your
  newsletter sender list, sample emails for prompt tuning, and the
  Airtable base structure. Deliverable from the call is a written
  recommendation + a firm price for Phases 2 + 3.

  Weeks 2–3 — Phase 2 (daily pipeline) on the live Gmail feed +
  Airtable base. Stops at: "5 new newsletters come in today, AI-
  curated tips land in the Review Dashboard tomorrow, approved
  items move to the Master Library on checkbox tick."

  Weeks 4–5 — Phase 3 (historical sweep) via the Python +
  Anthropic batch API path. Estimate is for cost-only ($50–100); my
  time on this is the script + the verification, not per-email.

  Week 6 — Handover (runbook, 5-min Loom, prompt-rotation
  walkthrough) + 14 days of defect-fix support.

Pricing: hourly USD 95 for Phase 1 (probably 6–10 hrs); fixed-price
quotes for Phases 2 + 3 delivered as written deliverable from the
scoping call (so you see the firm number before you commit).

Two questions to keep this short:
  1. How many initial clusters are you launching with — the 3 you
     named (AI, Finance, Health) or are there more queued?
  2. The "Approval" trigger — do you want me to use Relay.app
     (which you mentioned) or wire it straight into Airtable
     automations? They both work; I'd default to Airtable native
     unless you have a specific Relay reason.

Talk soon,
Derek
```
